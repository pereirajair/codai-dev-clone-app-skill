# Stage 7: Fix Agent Reference

This file governs the QA↔Fix convergence loop. The model here is **measured, not eyeballed**. Read it against `00-contract.md` — §5 (the computed-style GATE), §6 (the convergence loop + escalation), §1 (exact paths), §7 (agent-prompt standard). Where this file and the contract disagree, **the contract wins**.

**The single most important thing to internalize:** the QA agent does NOT decide pass/fail and does NOT run a fixed number of cycles. The **orchestrator** computes PASS from `clone-styles.json` + `assert-styles.mjs` (computed-style assertions vs the design tokens) plus a clean `npm run build`, loops until converged, and handles escalation. There is no "minimum 4 cycles", no "0 F and ≤3 C" verdict, and no "open both screenshots and compare with your eyes" pass/fail step. Those are gone. Your job is to read the clone's **computed styles** off the live localhost page, write them to a JSON file, and turn each assertion failure into a measured bug fragment.

**Why computed-style assertions, not file pixel-diff?** This skill runs in-conversation through the **Claude Chrome extension** (`mcp__claude-in-chrome__*`). The extension reads the live DOM and can run JS to pull `getComputedStyle` for any selector, but it does **not** write screenshot PNGs to arbitrary workspace paths the way a CLI screenshot tool would — so there is no saved-file pixel diff to gate on. The objective gate is therefore the **computed-style assertion suite**: every selector's measured value vs the `DESIGN.md` / `assertions.json` token. A screenshot is still taken via the extension as a **visual reference image** the human/agent eyeballs against the real site — it's a visual aid, never a saved-file pixel comparison.

## How the loop actually runs (orchestrator-owned — context, not your job)

You are one fan-out task inside this machine. The orchestrator drives it; you never make the looping decision.

```
start_dev_server (poll localhost:3000 until ready; else terminal HARD-BLOCKER)
cycle = 0; prev_failed = 999; stall = 0; MAX_CYCLES = 10
while cycle < MAX_CYCLES:
  cycle++
  QA   → fan out one task per page × viewport             (THIS = Stage 6, you)
       → each task navigates the clone with the Chrome extension,
         reads computed styles via javascript_tool,
         writes 06-qa/cycle-{N}/clone-styles.json          (the GATE input, §5)
       → orchestrator runs `node scripts/assert-styles.mjs`
         comparing clone-styles.json against 03-design-spec/assertions.json + DESIGN.md
  if GATE PASS: → terminal CONVERGED-PASS; break
  failed = style_assertions.failed (across all page × viewport)
  stall = (prev_failed - failed < 1) ? stall+1 : 0;  prev_failed = failed
  if cycle ≥ 2 and stall ≥ 2: → terminal STUCK; break
  Fix  → partition bugs into disjoint file-sets, fan out  (THIS = Stage 7)
       → re-read computed styles for each affected selector after each change; revert if not improved
  npm run build  (regression gate — must exit 0)
if cycle == MAX_CYCLES: → terminal CEILING
```

**The GATE (§5), computed by the orchestrator — ALL must hold:**
`style_assertions.failed == 0` AND `npm run build` exits 0. (Live thresholds / the assertion set: `00-config.json` → `gate` and `03-design-spec/assertions.json`.)

**Terminal states (when control returns to the human):** `CONVERGED-PASS` · `STUCK` · `HARD-BLOCKER` (auth wall / login redirect / build won't compile) · `CEILING`. You never emit these; the orchestrator does. You only emit `<promise>CONTINUE</promise>` on success or `<promise>BLOCKED: reason</promise>` if the page is unreachable.

Severity labels (A/B/C/F) you put on bugs are **triage signal for the Fix agent's ordering**, not a pass/fail verdict. A clone can be CONVERGED-PASS with open B/C bugs only if every assertion in the gate set passes; it FAILS while a single asserted selector's computed value still drifts from its token.

# QA Agent (Stage 6)

**IMPORTANT:** All browser work uses the **Claude Chrome extension** (`mcp__claude-in-chrome__*`). The clone has no auth, so the extension navigates straight to `http://localhost:3000{route}`. You read the clone's computed styles with `mcp__claude-in-chrome__javascript_tool` and capture a reference screenshot with `mcp__claude-in-chrome__computer`. (Chrome extension tools must be loaded via `ToolSearch` — `select:mcp__claude-in-chrome__navigate`, etc. — before they can be called.) **One Chrome, one view at a time** — QA capture is sequential across page×viewport even though the *analysis* fans out.

You are the QA agent for **ONE page at ONE viewport** for **ONE cycle**. You have NEVER seen the build process and you have NOT read the application source. Fresh context. You are the most critical agent in the pipeline: the gate is only as honest as the computed styles you read and the tokens you assert against. "Close enough" is not a thing here — there is a number for every assertion.

## Inputs (exact paths — §1)

| Source | What it gives you |
|--------|------------------|
| Running dev server at `http://localhost:3000{route}` | the clone to read computed styles from |
| `03-design-spec/assertions.json` | the selector → property → expected-token assertion set (the gate's targets) |
| `03-design-spec/DESIGN.md` | the authoritative tokens — exact hex/px/weight behind each assertion |
| `01-recon/screenshots/{page-slug}--{viewport}.png` | the real-site reference shot — what your extension screenshot should visually match |

Read `DESIGN.md` and `assertions.json` before you author any bug. They carry the exact expected values (hex, px, weight, easing) — never approximate when the token is sitting right there.

## Determinism hygiene — do this BEFORE you read styles / screenshot, or your evidence is garbage

Computed styles and the reference screenshot are only meaningful if the page is in its settled, resting state. Anything non-deterministic (a mid-flight animation, an unloaded webfont falling back to system, an unhydrated skeleton) gives a misread value or a misleading reference image. Eliminate every source of jitter first:

1. **Disable animations / transitions.** Emulate `prefers-reduced-motion: reduce` (inject a `* { transition: none !important; animation: none !important; }` style via `javascript_tool`, or set the media emulation) so transitions, CSS animations, and `@keyframes` are frozen at their resting state. You want the *resting* computed value, never a value sampled mid-transition.
2. **Wait for fonts AND hydration.** Block until `document.fonts.ready` resolves and the framework has hydrated (no skeleton/spinner left, layout settled) before reading `getComputedStyle`. A `fontFamily`/`fontWeight` read before the webfont swaps in will report the system fallback and flag every text run as a false failure. Give the page a settle beat after navigation; poll for the absence of loading shells rather than sleeping a fixed time.
3. **Ignore known-dynamic regions.** Live counters, relative timestamps ("2 min ago"), randomized hero imagery, ad slots — these are not assertion targets and never carry a stable computed value worth gating. `assertions.json` should not target them; if you spot a dynamic region someone added an assertion for, note it in the bug fragment as `evidence_method: "dynamic-region (excluded from gate)"` rather than filing it as a defect.
4. **Pin the viewport to the baseline's exact pixels** before navigating, so the layout reflows identically and your reference screenshot lines up with the recon shot. Viewport keys (must match recon exactly): `desktop` 1920×1080, `tablet` 768×1024, `mobile` 375×667 — set via `mcp__claude-in-chrome__resize_window`.

## Procedure (ONE page × ONE viewport)

### Step 1 — Navigate the clone and freeze it

```
mcp__claude-in-chrome__resize_window      → the {viewport}'s exact pixels (e.g. 1920 × 1080)
mcp__claude-in-chrome__navigate           → "http://localhost:3000{route}"
mcp__claude-in-chrome__javascript_tool    → inject prefers-reduced-motion / no-animation style (hygiene #1)
mcp__claude-in-chrome__javascript_tool    → await document.fonts.ready; poll until no skeletons/spinners (hygiene #2)
```

If the clone route 302s to `/login`, renders a blank/auth wall, or never reaches ready — do NOT read styles off a login screen and pretend it's the page. Emit `<promise>BLOCKED: {route} redirects to auth / unreachable</promise>` and stop. The orchestrator escalates this as a HARD-BLOCKER (§6); burning cycles on a login screen is forbidden.

### Step 2 — Read computed styles for every asserted selector → write `clone-styles.json`

Run `getComputedStyle` over **every selector in `assertions.json` that applies to this route**, via `mcp__claude-in-chrome__javascript_tool`. For each selector, read the asserted properties (at minimum `color`, `backgroundColor`, `backgroundImage`, `fontFamily`, `fontSize`, `fontWeight`, `letterSpacing`, `lineHeight`, `borderRadius`, `boxShadow`, `cursor`, `transition`) plus `getBoundingClientRect()` for geometry ground truth. Example JS shape to run:

```js
const out = {};
for (const sel of SELECTORS) {                // SELECTORS comes from assertions.json
  const el = document.querySelector(sel);
  if (!el) { out[sel] = null; continue; }     // missing element → record null (a real failure)
  const cs = getComputedStyle(el);
  out[sel] = {
    color: cs.color, backgroundColor: cs.backgroundColor, backgroundImage: cs.backgroundImage,
    fontFamily: cs.fontFamily, fontSize: cs.fontSize, fontWeight: cs.fontWeight,
    letterSpacing: cs.letterSpacing, lineHeight: cs.lineHeight,
    borderRadius: cs.borderRadius, boxShadow: cs.boxShadow,
    cursor: cs.cursor, transition: cs.transitionProperty + ' ' + cs.transitionDuration,
    rect: el.getBoundingClientRect().toJSON()
  };
}
return out;
```

Write the result to the EXACT path (the GATE input, §5):

```
06-qa/cycle-{N}/clone-styles.json
```

Shape: `{ "page": "...", "viewport": "...", "route": "...", "styles": { "<selector>": { ...computed... } | null } }`. A `null` for a selector that `assertions.json` expects is itself a measured failure (the element is absent) — record it, don't omit it.

### Step 3 — Capture a reference screenshot (visual aid, NOT a saved-file pixel diff)

```
mcp__claude-in-chrome__computer            → screenshot the frozen, settled clone
```

This screenshot is a **visual reference image**: you (or the human in guided mode) eyeball it side-by-side against the real-site shot at `01-recon/screenshots/{page-slug}--{viewport}.png` to catch gross layout/structure problems the per-selector assertions can't express (a whole section missing, blocks in the wrong order). It is **not** diffed as a file and produces **no** `mismatch_pct` and **no** `.diff.png` — the extension doesn't write comparable PNGs to the workspace, so there is nothing to pixel-diff. Treat it strictly as a human/agent sanity check; the **objective gate is the computed-style assertions in Step 2**.

### Step 4 — Let the orchestrator run the assertions

After `clone-styles.json` lands, the orchestrator runs:

```bash
node scripts/assert-styles.mjs --cycle {N}
```

It compares `06-qa/cycle-{N}/clone-styles.json` against `03-design-spec/assertions.json` (which references `DESIGN.md` tokens) — colors normalized to `rgb()`, numerics ±1px / ±0.01em — and writes the gate result:

```
06-qa/cycle-{N}/style-report.json
```

```json
{ "cycle": N,
  "style_assertions": {
    "total": 142, "passed": 138, "failed": 4,
    "failures": [
      {"page":"home","viewport":"desktop","selector":".hero-title","prop":"fontWeight","expected":"800","actual":"700"}
    ] } }
```

You do not run `assert-styles.mjs` yourself and you do not compute pass/fail. You read the failures whose `selector` resolves on your page and turn each one into a measured bug fragment.

### Step 5 — Author measured bug fragments from the assertion failures

For each assertion failure on your page×viewport:

1. **Localize it.** Map the failing `selector` to the element/section on the page (use `getBoundingClientRect`, the design-system selectors, and the DOM structure). The reference screenshot helps you confirm *where* it sits.
2. **Measure both sides.** The clone value is the `actual` you already read into `clone-styles.json` (re-read with `getComputedStyle` if you need to confirm). The expected value is the `DESIGN.md` token behind that assertion. The bug is the *delta between two measured numbers* — never an impression.
3. **Pick the evidence method.** EVERY bug carries an `evidence_method` that is a real measurement: the failing assertion itself — `getComputedStyle` value vs the design-system token (e.g. `"getComputedStyle('.hero-title').fontWeight = '700'; token --font-weight-display = 800"`). No bug ships with "looks off" — if it isn't a measured computed-style delta (or a `null`/absent element), you can't file it.
4. **Name the target file** in `file` (and echo it in `fix_hint`) so the orchestrator can partition bugs into disjoint file-sets for parallel fixing. Be specific: component path, the property, the token to apply.
5. **Grade severity for triage.** A/B/C/F is the Fix agent's ordering signal, not a verdict:
   - **F** — asserted element absent (`null`), wrong color family, or a structural break the reference screenshot also reveals.
   - **C** — clearly wrong shade / wrong weight / ≳5px geometry drift on an asserted property; an unmistakable token violation.
   - **B** — small deltas (1–2 shades, sub-5px) that still trip the assertion tolerance.
   - **A** — within tolerance; only file to record a near-miss. Don't manufacture A entries.
   When unsure between two grades, grade **worse**.

Write the fragment to the EXACT path (parallel-safe — never write a shared file):

```
06-qa/cycle-{N}/fragments/{page-slug}--{viewport}.bugs.json
```

Shape — a JSON array; the orchestrator merges fragments and renumbers ids into `bugs.json` (do NOT assign global ids yourself):

```json
[
  {
    "severity": "C",
    "element": "Hero title",
    "selector": ".hero-title",
    "viewport": "desktop",
    "expected": "fontWeight 800 (token --font-weight-display)",
    "actual": "fontWeight 700",
    "evidence_method": "getComputedStyle('.hero-title').fontWeight = '700' vs DESIGN.md --font-weight-display = 800 (assert-styles failure)",
    "fix_hint": "Set hero title to font-weight 800 / font-display class; not 700",
    "file": "app/components/Hero.tsx"
  },
  {
    "severity": "F",
    "element": "Primary CTA background",
    "selector": "button.cta-primary",
    "viewport": "desktop",
    "expected": "linear-gradient(135deg,#6d28d9,#9333ea) (token --grad-cta)",
    "actual": "flat #6d28d9 (no gradient)",
    "evidence_method": "getComputedStyle('button.cta-primary').backgroundImage = 'none'; token --grad-cta present in DESIGN.md (assert-styles failure)",
    "fix_hint": "Apply --grad-cta gradient to .cta-primary background-image",
    "file": "app/components/Cta.tsx"
  }
]
```

`evidence_method` is **required and must contain a measured computed-style value (or a `null`/absent element).** A fragment with an evidence-free bug is an invalid fragment.

### Completion

Write your `clone-styles.json`, your reference screenshot (kept in the session as the visual aid), and your bug fragment to their exact paths, set this task's flag in `status.json`, and end with `<promise>CONTINUE</promise>` (or `<promise>BLOCKED: reason</promise>` if the route was unreachable). Do not write `style-report.json`, do not write a pass/fail verdict, do not write a `qa-report.md` — those are orchestrator-owned.

## QA Agent Rules

- You do NOT decide pass/fail and you do NOT decide whether another cycle runs. The orchestrator runs `assert-styles.mjs` and reads the GATE (§5). Never soften or invent grades to influence the loop.
- Read computed styles at the recon baseline's exact viewport, with animations frozen and fonts/hydration settled — otherwise the values are noise, not signal.
- Every bug carries a measured `evidence_method` (a `getComputedStyle` value vs token, or a `null`/absent element). Never file "looks off." Never assert a match without a number.
- The extension screenshot is a **visual reference aid only** — eyeball it against the recon shot for gross structural problems; it is never a saved-file pixel diff and produces no mismatch %.
- Never read the application source to infer intent — measure the running browser and compare to `DESIGN.md` / `assertions.json`. (You may name a `file`/`fix_hint` from the component map, but the *defect* is a measured delta.)
- If a value can't be read or the page can't be reached, record `null` + why, or emit `<promise>BLOCKED: reason</promise>`. Never generate findings from memory.
- One `clone-styles.json` + one fragment per page×viewport, at the exact paths. Never write a shared file; the orchestrator does all merging.

# Fix Agent (Stage 7)

You are a Fix agent owning **ONE disjoint set of files** (`{FIXSET}`). You run in parallel with other Fix agents, each owning a different file-set. The cardinal rule: **edit ONLY files in your set.** Touching another set's files corrupts a sibling's work and breaks the parallel guarantee. You **edit code, you do not drive a browser to capture** — your re-check is a computed-style re-read (see step 3). You do not re-grade or re-author bugs — that's the QA agent's job. You make the assertions pass and prove it with a number.

## Inputs (exact paths — §1)

| Source | What it gives you |
|--------|------------------|
| `{FIXSET}` | your partition — a JSON array of bugs whose `file` is in your set |
| `06-qa/cycle-{N}/style-report.json` | current per-selector assertion failures — your baseline-to-beat |
| `06-qa/cycle-{N}/clone-styles.json` | the clone's measured computed values per selector |
| `03-design-spec/assertions.json` | the selector → property → expected-token assertions |
| `03-design-spec/DESIGN.md` | the exact token values to apply |

## Fix Procedure

### 1. Order by assertion weight, not by severity label

Sort your bugs by **how many assertions each fix clears** (and how visible the region is — confirm against the reference screenshot if you have it). A single change that flips a whole section's color/gradient back to token clears more of the gate than ten 1px radius nits. Severity (A/B/C/F) is a tiebreaker. The goal is to drive `style_assertions.failed` to 0 fastest.

### 2. Fix to the token

For each bug, in order:
1. Read its `fix_hint`, `file`, `selector`, and `expected` token value.
2. Apply the **minimal** change that brings the computed value to the `DESIGN.md` token. Do not refactor, do not restyle adjacent elements, do not touch routing/mock-data/build config unless the bug explicitly requires it.

### 3. Re-check after EACH change — re-read computed styles, revert if assertions didn't improve

This is the non-negotiable discipline that makes the loop converge instead of oscillate. The Fix agent does not capture screenshots; it re-reads the computed style for the affected selector(s) and re-runs the assertion check:

```bash
# after the edit, re-read the computed style for the affected selector(s) and re-assert:
node scripts/assert-styles.mjs --page {page-slug} --viewport {viewport} --selector "{selector}"
# (this re-reads getComputedStyle for the targeted selector(s) on the running clone and
#  recomputes pass/fail against assertions.json — same determinism hygiene as QA:
#  prefers-reduced-motion frozen, document.fonts.ready settled, baseline viewport)
```

- If the targeted assertion(s) now **pass** (failed count for those selectors **dropped**) → keep the change, move on.
- If they **did not improve** (still failing, or you broke a sibling assertion) → **REVERT** the change immediately and try a different approach. A fix that doesn't move the assertion count is not a fix; keeping it pollutes the next QA cycle and can stall convergence into a STUCK terminal (§6).

Same determinism hygiene as QA applies to the re-read (frozen animations, fonts/hydration settled, baseline viewport) — otherwise you can't trust whether the computed value actually moved to the token.

### 4. Keep the build compiling

After your changes, the build must stay green:

```bash
npm run build   # must exit 0
```

If an edit introduces a type/build error, fix it within your file-set or revert the offending change. A broken build is a regression gate failure (§6) and blocks the whole cycle. Never leave the tree non-compiling.

## Fix Agent Rules

- Edit ONLY files in `{FIXSET}`. Never touch another set's files — that's a parallel-safety violation.
- Order by how many assertions each fix clears first; severity label is only a tiebreaker.
- Apply the `DESIGN.md` token, not a guess; use the failing assertion's `selector`/`expected` to target the exact element.
- Re-read computed styles for the affected selector(s) after EVERY change; if the assertion didn't pass, REVERT and try again. No "I think it's better" — the assertion result decides.
- Make minimal, targeted edits — no refactors, no unrelated restyling, no routing/mock-data/build-config changes unless the bug demands it.
- Keep `npm run build` green at all times.
- You do not re-grade, do not author new bugs, do not write `style-report.json`. End with `<promise>CONTINUE</promise>`.
